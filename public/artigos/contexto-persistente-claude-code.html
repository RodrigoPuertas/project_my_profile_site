Contexto Persistente
no Claude Code

Como construir um sistema de memória em arquivos markdown para o Claude Code nunca mais esquecer seus projetos, decisões e histórico entre sessões.

O problema

O Claude Code reseta o contexto a cada nova sessão. Isso significa que toda vez que você abre o editor, ele não sabe quem é você, quais projetos estão em andamento, quais decisões técnicas foram tomadas, nem o que foi feito na última sessão.

O fluxo sem nenhum sistema é sempre o mesmo: você abre o projeto, começa a digitar, e inevitavelmente precisa passar cinco minutos reexplicando contexto antes de conseguir trabalhar de verdade.

O custo real: não é só tempo perdido na explicação. É a degradação da qualidade das respostas — sem contexto completo, o Claude toma decisões genéricas que ignoram restrições e decisões anteriores do seu projeto.

A solução não precisa de ferramentas externas, APIs ou configurações complexas. O Claude Code já tem um mecanismo nativo para isso: ele lê automaticamente arquivos chamados CLAUDE.md antes de qualquer sessão. O que falta é estruturar bem esses arquivos e automatizar o log de sessões.

A arquitetura

O sistema funciona em dois níveis: um contexto global sobre quem você é e como trabalha, e um contexto por projeto com stack, arquitetura e decisões específicas. 

~/.claude/
├── CLAUDE.md                    # perfil global — lido em toda sessão
├── settings.json                # hook de log automático
├── scripts/
│   └── log-session.ps1          # script do hook
└── memory/
    ├── projects.md              # status de cada projeto ativo
    ├── sessions.md              # log do que foi feito
    └── decisions.md             # decisões técnicas para não revisitar

cada-projeto/
└── CLAUDE.md                    # contexto específico do projeto

Tudo é markdown puro. Sem banco de dados, sem dependências, sem serviços externos. Versionável no Git, editável em qualquer editor.

Passo 1 — CLAUDE.md global

O arquivo ~/.claude/CLAUDE.md é lido pelo Claude Code automaticamente em toda sessão, independente do projeto aberto. É onde você define quem é, como gosta de trabalhar e as regras de comportamento. 

# Seu Nome — Contexto Global

## Quem sou eu
- Sua área de atuação e empresa
- O que está construindo

## Stack principal
- Linguagens, ferramentas, infra

## Como trabalho
- Idioma e nível técnico preferido
- Regras de abordagem

## Regras para o Claude
1. Nunca peça para reexplicar o que já está em memory/ — leia primeiro
2. Ao iniciar sessão: ler memory/projects.md e memory/sessions.md
3. Ao encerrar sessão: atualizar memory/sessions.md
4. Se houver CLAUDE.md no projeto atual: lê-lo antes de qualquer ação

A regra mais importante é a número 1. Sem ela, o Claude vai perguntar sobre coisas que já estão documentadas nos arquivos de memória.

Passo 2 — Estrutura de memória

Três arquivos na pasta ~/.claude/memory/ cobrem as informações que mais se perdem entre sessões.

projects.md

Status de cada projeto ativo. Atualizado sempre que o estado de um projeto muda.

## Nome do Projeto
- **Stack:** Python, FastAPI, PostgreSQL
- **Status:** Em andamento — fase de testes de integração
- **Próximos passos:**
  - [ ] Implementar endpoint de autenticação
  - [ ] Configurar CI no GitHub Actions
- **Repositório:** github.com/user/projeto
- **Deadline:** 2026-06-30

sessions.md

Log do que foi feito em cada sessão. Substitui a necessidade de reexplicar histórico.

## 2026-05-15 — nome-do-projeto
**O que foi feito:**
- Implementei validação de schema no pipeline de ingestão
- Corrigi bug de encoding em dados UTF-8 do legado

**Decisões tomadas:**
- Usar Pydantic v2 em vez de marshmallow — melhor performance

**Próximos passos:**
- [ ] Adicionar testes unitários para o validador

decisions.md

Decisões técnicas importantes com contexto e trade-offs. Evita revisitar o mesmo debate em sessões futuras — ou pior, tomar a decisão oposta por falta de memória. 

## 2026-05-10 — Não usar calibração isotônica em produção
**Contexto:** modelo LightGBM com calibração melhorava precision mas reduzia recall
**Decisão:** manter modelo sem calibração
**Motivo:** objetivo é capturar máximo de no-shows — recall é prioritário
**Trade-offs:** precision menor, mais falsos positivos aceitáveis

Passo 3 — Hook de log automático

O ponto mais importante para manter o sistema vivo: automatizar o lembrete de atualizar o log. Sem automação, o sistema morre na segunda semana.

O Claude Code suporta hooks — comandos que executam em eventos específicos. O evento Stop dispara toda vez que o Claude encerra uma resposta. Usamos isso para escrever um lembrete automático no sessions.md.

Atenção ao escaping: não escreva o comando PowerShell inline no JSON do settings.json. O escaping duplo (JSON + PowerShell) é frágil e difícil de manter. Sempre extraia para um .ps1 separado.

O script

Crie ~/.claude/scripts/log-session.ps1:

$date = Get-Date -Format 'yyyy-MM-dd HH:mm'
$path = "$env:USERPROFILE\.claude\memory\sessions.md"
$entry = @"

<!-- session-auto -->
## $date — sessao encerrada
**Lembrete:** atualize este log com o que foi feito.

"@
Add-Content -Path $path -Value $entry

O hook no settings.json

Adicione o bloco hooks em ~/.claude/settings.json:

{
  "hooks": {
    "Stop": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "powershell -ExecutionPolicy Bypass -File \"%USERPROFILE%\\.claude\\scripts\\log-session.ps1\""
          }
        ]
      }
    ]
  }
}

Testando

Antes de depender do hook, teste manualmente:

powershell -ExecutionPolicy Bypass -File "$env:USERPROFILE\.claude\scripts\log-session.ps1"

Verifique se uma entrada apareceu no final de sessions.md. Se sim, o hook vai funcionar.

Passo 4 — CLAUDE.md por projeto

Cada repositório deve ter um CLAUDE.md na raiz. O Claude Code detecta e lê automaticamente quando você abre o projeto no VSCode. Aqui vai o contexto que seria muito específico para ficar no arquivo global. 

# Nome do Projeto

## Contexto
- **Objetivo:** descrição clara e direta
- **Stack:** Python 3.11, FastAPI, PostgreSQL 15
- **Deadline:** 2026-06-30

## Arquitetura
# Descreva como o sistema está organizado
# Não precisa ser exaustivo — só o suficiente para orientar decisões

## Decisões técnicas
# Decisões específicas deste projeto que não devem ser revisitadas

## O que NÃO fazer
# Armadilhas conhecidas, tentativas que não funcionaram

A seção "O que NÃO fazer" é subestimada. Ela evita que o Claude repita abordagens que você já tentou e descartou — especialmente em projetos mais longos onde é fácil esquecer por que certas decisões foram tomadas.

Uso no dia a dia

Com o sistema montado, a rotina é simples. Você abre o projeto no VSCode, o Claude já carregou o contexto global e o do projeto. Trabalha normalmente. Ao encerrar, o hook dispara o lembrete — você substitui pelo log real do que foi feito naquela sessão.

Para adicionar um novo projeto: copie o template para a raiz do repo como CLAUDE.md, preencha os campos, adicione uma entrada em projects.md. Leva menos de cinco minutos e economiza muito mais do que isso em cada sessão futura.

O que este sistema não é: não é automágico. O Claude não atualiza os arquivos sozinho — ele lembra de tudo que está escrito, mas a curadoria do que merece ser registrado ainda é sua. A disciplina de atualizar o log após sessões produtivas é o que mantém o sistema útil.

O que vem a seguir

Com o contexto persistente funcionando, o próximo passo natural é criar comandos customizados no Claude Code — slash commands como /status para ver projetos ativos ou /log para estruturar o log da sessão atual sem sair do editor.

Outro caminho é automatizar ainda mais o log usando hooks mais específicos por tipo de operação — commit hooks que registram o que foi alterado, por exemplo. Mas isso já é otimização. O sistema descrito aqui resolve o problema central: nunca mais começar uma sessão do zero.
