O que é Data Engineering?

Pipelines, ETL, Data Lakes e Warehouses: a infraestrutura que sustenta toda análise de dados.

O que faz um Data Engineer?

Data Engineering (Engenharia de Dados) é a disciplina responsável por projetar, construir e manter a infraestrutura que permite a coleta, o armazenamento, a transformação e a disponibilização de dados em escala. Sem ela, cientistas de dados não têm dados limpos para modelar, analistas não têm tabelas confiáveis para consultar, e dashboards mentem.

É a fundação invisível: quando funciona, ninguém percebe. Quando falha, tudo desmorona.

Definição técnica: Engenharia de Dados é a prática de projetar e operar sistemas que coletam dados brutos de diversas fontes, transformam esses dados em formatos confiáveis e os disponibilizam para consumo por equipes de analytics, ciência de dados e produto.

Data Engineer vs Data Scientist vs Data Analyst

Essas três funções se complementam, mas têm escopos distintos:

  • Data Engineer (DE) — Constrói e mantém os pipelines e a infraestrutura. Foco em confiabilidade, escalabilidade e performance. Ferramentas: SQL, Spark, Airflow, dbt, Kafka.
  • Data Scientist (DS) — Cria modelos preditivos e algoritmos de Machine Learning. Consome dados preparados pelo DE. Ferramentas: Python, Scikit-learn, PyTorch, notebooks.
  • Data Analyst (DA) — Extrai insights de negócio a partir dos dados. Cria dashboards e relatórios. Ferramentas: SQL, Excel, Tableau, Power BI, Looker.

O Data Engineer é quem garante que os dados cheguem limpos, no formato certo, no lugar certo, no tempo certo. Sem esse trabalho, DS e DA ficam parados.

ETL vs ELT

ETL e ELT representam duas abordagens para mover dados de sistemas de origem para sistemas de destino. A diferença está na ordem em que a transformação acontece.

ETL — Extract, Transform, Load

A abordagem tradicional. Os dados são extraídos da fonte, transformados em um servidor intermediário (muitas vezes on-premise) e depois carregados no destino. Era o padrão antes da nuvem dominar.

  • Vantagem — Dados chegam já limpos e estruturados no destino
  • Desvantagem — Transformação é bottleneck; difícil escalar; reprocessamento caro

ELT — Extract, Load, Transform

A abordagem moderna. Os dados brutos são extraídos e carregados diretamente no destino (geralmente um Data Warehouse na nuvem como BigQuery, Snowflake ou Redshift), e a transformação acontece lá dentro, aproveitando o poder computacional do warehouse.

  • Vantagem — Escalabilidade nativa; dados brutos preservados; reprocessamento fácil
  • Desvantagem — Requer governança forte para não virar um "lixão de dados"

Arquitetura de Dados Moderna

Uma arquitetura de dados moderna segue um fluxo claro: dados nascem em diversas fontes, passam por uma camada de ingestão, são armazenados em um Data Lake com camadas progressivas de refinamento, migram para um Data Warehouse otimizado para consultas analíticas, e finalmente alimentam dashboards, relatórios e modelos de Machine Learning.

O ponto central dessa arquitetura é que cada camada tem uma responsabilidade clara. A ingestão traz os dados; o Data Lake armazena com fidelidade; o Warehouse otimiza para consultas; e a camada de consumo entrega valor.

Ferramentas essenciais do Data Engineer

O toolkit de um engenheiro de dados moderno e vasto, mas algumas ferramentas se tornaram indispensaveis:

SQL

A língua franca dos dados. Todo pipeline, todo warehouse, toda transformação passa por SQL. Não importa se você usa Spark, dbt ou BigQuery — SQL está no centro.

Apache Spark / PySpark

O motor de processamento distribuído mais utilizado no mundo. Processa terabytes de dados em paralelo usando clusters. PySpark é a interface Python que torna o Spark acessível para quem vem de data science.

Apache Airflow

O orquestrador de pipelines mais popular. Permite definir DAGs (Directed Acyclic Graphs) em Python que representam fluxos de trabalho. Cada task é um passo do pipeline, e o Airflow gerencia dependências, agendamento, retries e alertas.

dbt (Data Build Tool)

Revolucionou a transformação de dados. Permite escrever transformações em SQL puro, versionar no Git, testar automaticamente e documentar. É o "T" do ELT moderno.

Plataformas de Cloud

  • AWS — S3, Glue, Redshift, EMR, Kinesis
  • GCP — GCS, BigQuery, Dataflow, Dataproc, Pub/Sub
  • Azure — ADLS, Synapse, Data Factory, Databricks

A maioria das empresas usa pelo menos um desses ecossistemas. Saber navegar entre serviços de storage, compute e orquestração na nuvem é fundamental.

Ferramentas de ingestão

  • Apache Kafka — Streaming de eventos em tempo real
  • Airbyte / Fivetran — Ingestão de dados de SaaS e APIs
  • Debezium — Change Data Capture (CDC) de bancos relacionais

Qualidade de dados

De nada adianta ter pipelines rápidos e infraestrutura robusta se os dados que fluem são inconsistentes, duplicados ou incompletos. Qualidade de dados é um dos maiores desafios — e uma das maiores responsabilidades — do Data Engineer.

Dimensões de qualidade

  • Completude — Os dados estao completos? Ha campos nulos inesperados?
  • Unicidade — Existem registros duplicados?
  • Consistencia — Os dados concordam entre diferentes fontes?
  • Freshness (Frescor) — Os dados estao atualizados? Quanto tempo entre o evento e a disponibilizacao?
  • Conformidade — Os dados respeitam o schema e as regras de negocio esperadas?
  • Acuracia — Os dados refletem a realidade?

Data Contracts

Data Contracts são acordos formais entre produtores e consumidores de dados. Eles definem o schema, a frequência de atualização, as garantias de qualidade e o SLA esperado. É como uma API contract, mas para dados.

Na prática: Um data contract pode ser um arquivo YAML no repositório que define: nome do dataset, owner, schema (colunas + tipos), testes de qualidade obrigatórios, SLA de freshness e canal de comunicação para incidentes.

Ferramentas de qualidade

  • Great Expectations — Framework open-source para testes de dados
  • dbt tests — Testes de unicidade, nao-nulo, relacionamentos e custom SQL
  • Monte Carlo / Elementary — Observabilidade de dados (data observability)
  • Soda — Monitoramento e validacao de qualidade

Pipeline na prática

Vamos construir um exemplo simplificado de pipeline ELT que extrai dados de uma API, carrega em um Data Lake e transforma usando SQL.

1. Extração e carga (Python)


# extract_and_load.py
# Extrai dados de uma API e carrega no Data Lake (S3)
import requests
import boto3
import json
from datetime import datetime

def extract_orders():
    """Extrai pedidos da API de vendas."""
    response = requests.get(
        "https://api.example.com/orders",
        params={"date": datetime.today().strftime("%Y-%m-%d")}
    )
    return response.json()

def load_to_s3(data, bucket, key):
    """Carrega dados brutos no S3 (camada raw)."""
    s3 = boto3.client("s3")
    s3.put_object(
        Bucket=bucket,
        Key=key,
        Body=json.dumps(data),
        ContentType="application/json"
    )

# Execução
orders = extract_orders()
today = datetime.today().strftime("%Y/%m/%d")
load_to_s3(
    data=orders,
    bucket="data-lake-raw",
    key=f"orders/{today}/orders.json"
)
print(f"Loaded {len(orders)} orders to S3")

2. Transformação (SQL com dbt)


-- models/marts/fct_orders.sql
-- Modelo dbt: transforma pedidos brutos em fato de vendas

WITH raw_orders AS (
    SELECT *
    FROM {{ ref('stg_orders') }}
),

enriched AS (
    SELECT
        o.order_id,
        o.customer_id,
        o.order_date,
        o.total_amount,
        o.status,
        c.customer_segment,
        c.acquisition_channel,
        -- Métricas derivadas
        CASE
            WHEN o.total_amount >= 500 THEN 'high_value'
            WHEN o.total_amount >= 100 THEN 'medium_value'
            ELSE 'low_value'
        END AS order_tier,
        ROW_NUMBER() OVER(
            PARTITION BY o.customer_id
            ORDER BY o.order_date
        ) AS order_sequence
    FROM raw_orders o
    LEFT JOIN {{ ref('dim_customers') }} c
        ON o.customer_id = c.customer_id
    WHERE o.status != 'cancelled'
)

SELECT * FROM enriched

3. Orquestração (Airflow DAG)


# dags/orders_pipeline.py
from airflow import DAG
from airflow.operators.python import PythonOperator
from airflow.operators.bash import BashOperator
from datetime import datetime, timedelta

default_args = {
    "owner": "data-eng",
    "retries": 2,
    "retry_delay": timedelta(minutes=5),
}

with DAG(
    dag_id="orders_pipeline",
    schedule_interval="0 6 * * *",  # Todo dia as 6h
    start_date=datetime(2026, 1, 1),
    default_args=default_args,
    catchup=False,
) as dag:

    extract = PythonOperator(
        task_id="extract_orders",
        python_callable=extract_orders,
    )

    load = PythonOperator(
        task_id="load_to_s3",
        python_callable=load_to_s3,
    )

    transform = BashOperator(
        task_id="dbt_run",
        bash_command="dbt run --models fct_orders",
    )

    test = BashOperator(
        task_id="dbt_test",
        bash_command="dbt test --models fct_orders",
    )

    extract >> load >> transform >> test

Esse pipeline roda diariamente: extrai dados da API, carrega no S3 (camada raw), transforma com dbt no warehouse e executa testes de qualidade automaticamente.
