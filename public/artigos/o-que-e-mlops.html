O que é MLOps?

Como levar modelos de ML do Jupyter para produção com qualidade, rastreabilidade e escalabilidade.

O que é MLOps

MLOps (Machine Learning Operations) é o conjunto de práticas, ferramentas e cultura que une Machine Learning, Engenharia de Software e Operações (Ops) para automatizar e padronizar todo o ciclo de vida de modelos de ML em produção.

O termo surgiu como uma extensão natural do DevOps para o mundo de dados. Assim como DevOps resolveu o conflito histórico entre desenvolvedores e times de infraestrutura, MLOps resolve o conflito entre cientistas de dados que criam modelos em notebooks e engenheiros que precisam manter esses modelos rodando de forma confiável em produção.

Definição objetiva: MLOps é a disciplina de engenharia que aplica princípios de DevOps â€” CI/CD, automação, monitoramento, infraestrutura como código â€” ao ciclo de vida completo de modelos de Machine Learning, desde o treinamento até o deploy e o monitoramento contínuo.

Se DevOps cuida de código â†’ build â†’ deploy â†’ monitoramento, MLOps cuida de dados â†’ treinamento â†’ validação â†’ deploy â†’ monitoramento â†’ retrain. A diferença fundamental é que no ML, além do código mudar, os dados também mudam â€” e isso introduz uma camada extra de complexidade.

O problema que MLOps resolve

Existe um gap enorme entre treinar um modelo num Jupyter Notebook e colocá-lo em produção de verdade. Segundo pesquisas da indústria, cerca de 87% dos modelos de ML nunca chegam Ã  produção. Por quê?

  • O notebook não é produção â€” código não-modular, dependências não versionadas, sem testes
  • Dados mudam silenciosamente â€” data drift e concept drift degradam performance sem aviso
  • Não há reprodutibilidade â€” ninguém sabe qual versão do modelo está rodando, com quais dados foi treinado, quais hiperparâmetros foram usados
  • Retrain é manual â€” alguém precisa lembrar de retreinar o modelo, comparar métricas, fazer deploy manualmente
  • Sem monitoramento â€” o modelo pode estar errando em produção por semanas e ninguém percebe

A metáfora: Imagine um chef que cria uma receita incrível na cozinha experimental, mas não tem como replicá-la na cozinha do restaurante. Não há receita escrita, os ingredientes mudam todo dia, e ninguém verifica se o prato continua bom. Isso é ML sem MLOps.

MLOps existe para fechar esse gap. Ã‰ a ponte que transforma experimentos em notebooks em sistemas de ML confiáveis, rastreáveis e escaláveis.

Os pilares do MLOps

1. Versionamento de modelos e dados

Assim como versionamos código com Git, precisamos versionar modelos (pesos, arquitetura, hiperparâmetros) e dados de treinamento. Ferramentas como DVC (Data Version Control) e MLflow permitem rastrear exatamente qual modelo foi treinado com quais dados e parâmetros. 


# Versionamento com DVC
dvc init
dvc add data/training_set.parquet
git add data/training_set.parquet.dvc .gitignore
git commit -m "v2: adiciona novos dados de março"
dvc push

2. CI/CD para Machine Learning

Continuous Integration e Continuous Delivery adaptados para ML. Toda vez que novo código ou novos dados entram no pipeline, o sistema automaticamente retreina o modelo, roda testes de qualidade (data validation, model validation) e, se tudo passar, faz deploy.

  • CI â€” testes unitários do código de feature engineering, validação de schema de dados, testes de integração do pipeline
  • CD â€” deploy automático após validação, canary releases, rollback automático se métricas degradarem
  • CT (Continuous Training) â€” retreino automático quando novos dados chegam ou quando drift é detectado

3. Monitoramento em produção

Diferente de software tradicional, modelos de ML podem "quebrar silenciosamente" â€” continuar retornando predições, mas com qualidade degradada. Monitorar ML em produção exige:

  • Data drift â€” a distribuição dos dados de entrada mudou?
  • Concept drift â€” a relação entre features e target mudou?
  • Métricas de performance â€” accuracy, latência, throughput
  • Alertas automáticos â€” notificação quando métricas cruzam thresholds

4. Reprodutibilidade

Cada execução do pipeline deve ser 100% reproduzível. Isso significa rastrear: versão do código, versão dos dados, hiperparâmetros, seed aleatória, versão das dependências (Docker), e ambiente de execução. Se algo der errado em produção, você precisa conseguir voltar no tempo e entender exatamente o que aconteceu.

O que muda em cada nível

  • Level 0 â†’ 1: Você sai de scripts soltos para um pipeline orquestrado. O retreino vira automático. Você começa a versionar experimentos.
  • Level 1 â†’ 2: Você adiciona CI/CD ao pipeline de ML. Testes automatizados validam dados e modelos antes do deploy. Monitoramento fecha o loop com retrain automático.

Quando implementar MLOps

Comece com MLOps quando: você tem pelo menos um modelo em produção, ou quase lá, sua equipe perde tempo fazendo deploy e retrain manual, e você não consegue responder com confiança "qual versão do modelo está em produção agora?"

Não comece com MLOps quando: você ainda está na fase de exploração e não sabe se ML resolve o problema. Nesse estágio, foque em validar a hipótese e provar valor. MLOps vem depois.

Checklist prático para começar

  • Versione tudo â€” código (Git), dados (DVC), experimentos (MLflow). Esse é o passo zero.
  • Separe treinamento de serving â€” o código que treina o modelo não deve ser o mesmo que serve predições. Crie um pipeline de treinamento e uma API de serving separados.
  • Automatize o retrain â€” mesmo que seja um cron job simples. O objetivo é que o retreino não dependa de alguém lembrar de fazer.
  • Monitore o básico â€” distribuição das features de entrada, latência das predições, volume de requests. Se algo mudar drasticamente, você precisa saber.
  • Documente o pipeline â€” um diagrama simples mostrando de onde vêm os dados, como o modelo é treinado, e como chega em produção.

# Exemplo: pipeline MLOps minimo com GitHub Actions
# .github/workflows/retrain.yml

name: Retrain Model
on:
  schedule:
    - cron: '0 6 * * 1'  # toda segunda 6h
  workflow_dispatch:

jobs:
  retrain:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
      - run: pip install -r requirements.txt
      - run: python src/train.py
      - run: python src/evaluate.py
      - run: |
          if [ "$(cat metrics/improved.txt)" == "true" ]; then
            python src/deploy.py
          fi
