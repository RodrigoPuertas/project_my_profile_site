PySpark na Prática

Processamento distribuído de dados em larga escala: quando e por que usar Spark ao invés de Pandas.

O que é Apache Spark

Apache Spark é um motor de processamento de dados distribuído, open-source, projetado para lidar com volumes massivos de dados com velocidade e tolerância a falhas. Criado na UC Berkeley em 2009, rapidamente se tornou o padrão da indústria para Big Data.

O diferencial do Spark em relação ao Hadoop MapReduce é o processamento in-memory: em vez de escrever resultados intermediários em disco a cada etapa, o Spark mantém os dados na memória RAM dos workers, sendo até 100x mais rápido para determinadas operações.

Definição: PySpark é a API Python para o Apache Spark. Permite que cientistas e engenheiros de dados usem a sintaxe familiar do Python para processar dados distribuídos em clusters com centenas de máquinas.

Spark vs Pandas: quando usar cada um

A pergunta fundamental: quando o Pandas não é suficiente? A resposta está no volume de dados e na arquitetura de processamento.

Arquitetura do Spark

Entender a arquitetura é fundamental para otimizar seus jobs. O Spark segue um modelo master-worker com três componentes principais.

  • Driver Program — coordena o job, mantém o SparkContext e distribui o trabalho
  • Cluster Manager — gerencia recursos (YARN, Kubernetes ou Standalone)
  • Workers/Executors — executam as tasks em paralelo, cada um com sua fatia de memória e CPU

PySpark: primeiros passos

A porta de entrada é o SparkSession — o ponto central para toda interação com o Spark.

from pyspark.sql import SparkSession # Criar a sessão spark = SparkSession.builder \ .appName("MeuJob") \ .config("spark.sql.shuffle.partitions", 200) \ .getOrCreate() # Ler dados df = spark.read.parquet("s3://bucket/dados/") # Verificar schema e contagem df.printSchema() print(f"Total de linhas: {df.count()}")

Diferente do Pandas, o Spark usa lazy evaluation: operações como filter(), select() e groupBy() não executam imediatamente. O Spark constrói um plano de execução otimizado e só processa quando você chama uma action como count(), show() ou write().

Operações essenciais

Select, Filter e GroupBy

# Select e rename df.select("order_id", "value", "region") # Filter df.filter(df.value > 100) df.filter((df.region == "SP") & (df.value > 50)) # GroupBy + agregações from pyspark.sql import functions as F df.groupBy("region").agg( F.count("order_id").alias("total_pedidos"), F.sum("value").alias("receita_total"), F.avg("value").alias("ticket_medio") ).orderBy(F.desc("receita_total"))

Joins

# Inner join pedidos.join(clientes, pedidos.cliente_id == clientes.id, "inner") # Left join com broadcast (tabela pequena) from pyspark.sql.functions import broadcast pedidos.join(broadcast(regioes), "region_id", "left")

Window Functions

from pyspark.sql.window import Window # Ranking por região w = Window.partitionBy("region").orderBy(F.desc("value")) df.withColumn("rank", F.row_number().over(w)) \ .filter(F.col("rank") <= 10) # Média móvel de 7 dias w7 = Window.partitionBy("region") \ .orderBy("date") \ .rowsBetween(-6, 0) df.withColumn("media_movel_7d", F.avg("value").over(w7))

Spark SQL e o Catalyst Optimizer

Você pode usar SQL puro no Spark, inclusive via engines como Trino/Presto que na Shopee Express são o padrão para queries ad-hoc sobre o Data Lake.

# Registrar DataFrame como tabela temporária df.createOrReplaceTempView("pedidos") # SQL puro resultado = spark.sql(""" SELECT region, COUNT(*) AS total, AVG(value) AS ticket_medio, SUM(value) AS receita FROM pedidos WHERE date >= '2026-01-01' GROUP BY region ORDER BY receita DESC """) resultado.show()

Otimização e performance

Usar PySpark é fácil. Usar bem exige entender como o Spark distribui e processa dados.

1. Particionamento

O número de partições define o paralelismo. Poucas partições = workers ociosos. Muitas = overhead de coordenação.

# Ver partições atuais df.rdd.getNumPartitions() # Reparticionar (shuffle) df = df.repartition(200, "region") # Coalesce (sem shuffle — só reduz) df = df.coalesce(10)

2. Cache e Persist

Se você usa o mesmo DataFrame mais de uma vez, faça cache para evitar reprocessamento.

# Cache em memória df.cache() # Persist com nível específico from pyspark import StorageLevel df.persist(StorageLevel.MEMORY_AND_DISK)

3. Broadcast Joins

Quando uma tabela é pequena (dimensão, lookup), use broadcast para evitar shuffle — a tabela inteira é enviada a cada worker.

4. Evite UDFs sempre que possível

UDFs (User Defined Functions) em Python são serializadas e executadas fora da JVM, o que mata a performance. Prefira funções nativas do pyspark.sql.functions.

Regra de ouro: se existe uma função em F. (pyspark.sql.functions), use-a ao invés de escrever uma UDF. A diferença de performance pode ser de 10x a 100x.

Quando usar PySpark

Use PySpark quando: seus dados excedem a memória de uma máquina, você precisa de processamento distribuído, ou seus pipelines rodam em ambiente de cluster (EMR, Databricks, Dataproc).

Use PySpark quando:

  • Volume de dados > 10 GB (limite prático do Pandas)
  • Pipelines que rodam em produção com schedule (Airflow + Spark)
  • Queries complexas sobre Data Lake (Parquet, Delta Lake, Iceberg)
  • Necessidade de integração com ecossistema Hadoop/Hive

Não use PySpark quando:

  • Dados cabem na memória — Pandas é mais simples e rápido para prototipação
  • Análise exploratória rápida em dataset pequeno
  • Overhead de configuração de cluster não justifica o volume
